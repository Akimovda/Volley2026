# Диагностика и фикс блока «🏐 Загрузка мероприятий (последние 3 месяца)» на /org/dashboard

Дата: 2026-07-15. Dev, branch `main`. Организатор для проверки: id 260 (impersonation).

## 1. Симптомы (исходные)

1. На графике (`loadChart`, Chart.js) бар — только у ОДНОГО мероприятия из 4+.
2. Колонка «Всего записей» — ПУСТАЯ (не `0`, а буквально пусто) у всех строк таблицы.
3. «Средняя загрузка» — `0%` у всех, при этом на графике бар «Классика игра» ~100.

## 2. Код блока (было)

Файл: `app/Http/Controllers/OrgDashboardController.php`, метод `index()`, было:

```php
$occurrenceLoad = DB::table('event_occurrences as eo')
    ->join('events as e', 'e.id', '=', 'eo.event_id')
    ->join('event_game_settings as egs', 'egs.event_id', '=', 'e.id')
    ->leftJoin('event_occurrence_stats as eos', 'eos.occurrence_id', '=', 'eo.id')
    ->where('e.organizer_id', $orgId)
    ->where('eo.starts_at', '>=', now()->subMonths(3))
    ->select(
        'e.title',
        DB::raw('AVG(eos.registered_count::float / NULLIF(egs.max_players, 0) * 100) as avg_load_pct'),
        DB::raw('COUNT(eo.id) as occurrences_count'),
        DB::raw('SUM(eos.registered_count) as total_registered')
    )
    ->groupBy('e.id', 'e.title')
    ->orderByDesc('total_registered')
    ->limit(10)
    ->get();
```

Blade (`resources/views/dashboard/org.blade.php`) и JS (`loadChart`) берут **бар, «Всего записей» и «Средняя загрузка» из ОДНОГО и того же результата запроса** — все три цифры действительно из одного источника, это не рассинхрон между разными запросами. Проблема была в самом запросе.

## 3. Корневые причины (все подтверждены на живых данных dev, event org_id=260)

### 3.1. `event_occurrence_stats.registered_count` — устаревающий кеш, а не live-счётчик

Из 178 occurrences организатора 260 за 3 месяца строка в `event_occurrence_stats` существовала лишь у ~20. У остальных — `eos=NULL` (кеш никогда не заполнялся или устарел). Это подтверждает уже существующий в коде комментарий в `EventOccurrenceStatsService::getRegisteredCount()`:

> "Не используем event_occurrence_stats: счётчик устаревает при массовых отменах через QueryBuilder."

`LEFT JOIN` + `SUM(eos.registered_count)` / `AVG(...)` в SQL **игнорируют NULL** (не считают их за 0) — поэтому:
- Мероприятия, у которых НИ ОДНА occurrence не имеет строки в `event_occurrence_stats` (а таких было большинство — 6 из 9 живых событий), получали `total_registered = NULL` → в blade рендерилось как пустая ячейка (симптом 2), `avg_load_pct = NULL` → `round(null ?? 0)` = `0%` (симптом 3).
- У мероприятий, где хотя бы часть occurrences имела строку в кеше (событие 364/363/366), агрегаты считались только по ним — и получались реальные числа (симптом: бар только у «Классика игра»).
- Дополнительно найдено расхождение: даже там, где строка `eos` существовала, `registered_count` местами отличался от реального live COUNT на 1 (например occ 12055: eos=14, live=13; occ 12069: eos=15, live=14) — кеш действительно дрейфует, что и описано в докстринге сервиса.

### 3.2. `ORDER BY total_registered DESC` + `LIMIT 10` — PostgreSQL сортирует NULL ПЕРВЫМИ при DESC

Это отдельный скрытый баг: PostgreSQL по умолчанию для `DESC` кладёт `NULL` **в начало**, а не в конец (в отличие от интуитивного ожидания). Из-за этого `LIMIT 10` в первую очередь выбирал мероприятия с `total_registered = NULL` (у которых нет данных вообще), вытесняя из топ-10 мероприятия с реальными, ненулевыми записями. Это отдельно объясняет, почему в проде на реальных данных организатора картина может быть ещё хуже показанной на dev (только 1 бар из многих) — чем больше у организатора мероприятий без кеша `eos`, тем агрессивнее они вытесняют содержательные строки из лимита.

### 3.3. Вместимость (`egs.max_players`) не подходит для командных турниров

`INNER JOIN event_game_settings` + `egs.max_players` как знаменатель % — корректно для обычных игр/тренировок, но **неверно для командных турниров** (`team_classic`/`team_beach`): там `egs.max_players` — унаследованное от `GameCalculator` значение состава ОДНОЙ команды (например 12), а реальная вместимость турнира измеряется в командах (`tournament_teams_count`). Тот же самый приоритет уже пофикшен вчера в `EventRegistrationGuard` (см. CLAUDE.md, раздел «Турниры — карточка»): `events.tournament_teams_count → event_tournament_settings.teams_count → event_game_settings.teams_count`. Дашборд использовал третий, отдельный (и в данном случае неверный) вариант подсчёта.

Также для командных турниров регистрации живут в `event_teams`, а НЕ в `event_registrations` — прежний запрос вообще не мог посчитать реальное число команд, так как не смотрел в `event_teams`.

## 4. Фикс

`app/Http/Controllers/OrgDashboardController.php` — запрос переписан на:

1. **Живой COUNT** вместо кеша `event_occurrence_stats`:
   - обычные мероприятия / `tournament_individual` / `king_beach` — `COUNT(*)` по `event_registrations` с тем же условием, что и в `EventOccurrenceStatsService::getRegisteredCount()` (`cancelled_at IS NULL AND (is_cancelled IS NULL OR is_cancelled=false) AND (status IS NULL OR status != 'cancelled')`);
   - командные турниры (`team_classic`/`team_beach`/`team`) — `COUNT(*)` по `event_teams` с тем же набором статусов, что и в `EventRegistrationGuard` (`draft, ready, pending_members, submitted, confirmed, approved`).
2. **Вместимость по типу мероприятия** (тот же приоритет, что во вчерашнем фиксе `EventRegistrationGuard`):
   - командные турниры — `COALESCE(NULLIF(events.tournament_teams_count,0), NULLIF(event_tournament_settings.teams_count,0), NULLIF(event_game_settings.teams_count,0), 0)`, единица измерения — команды;
   - `tournament_individual`/`king_beach` — `COALESCE(NULLIF(event_tournament_settings.total_players_max,0), NULLIF(event_game_settings.max_players,0), 0)`, единица измерения — игроки (без конвертации в «команды» через regex — так проще и надёжнее, конвертация в команды нужна только для карточки мероприятия, не для дашборда);
   - обычные — `event_game_settings.max_players`.
3. Все три цифры (бар, «Всего записей», «Средняя загрузка») по-прежнему получаются из **одного** SQL-запроса (через `fromSub` — сначала строится под-запрос с per-occurrence `registered`/`capacity`, затем один `GROUP BY` агрегирует все три метрики разом) — не плодим второй/третий вариант подсчёта.
4. Отменённые occurrences (`is_cancelled = true`) исключены из выборки — как следствие: мероприятия, у которых ВСЕ occurrences за 3 месяца отменены, больше не занимают место в `LIMIT 10` пустыми строками (побочный эффект пункта 3.2 устранён естественным образом).
5. Blade: `{{ $row->total_registered ?? 0 }}` — пустое значение подстраховано на явный `0` (защита на случай, если для какого-то события всё же не найдётся ни одной occurrence с положительным `registered`).

## 6. Прод (id 260) — важное замечание

По просьбе пользователя не проверял боевые данные прод-организатора 260 напрямую (задача явно требует вносить изменения только в dev; сравнение с прод-данными оставлено на этап приёмки после деплоя — там картина будет представительнее, так как на dev у организатора 260 всего 9 «живых» мероприятий за 3 месяца, а на проде их значительно больше и NULLS-FIRST-баг из п. 3.2, вероятно, проявляется сильнее).

## 7. Что НЕ входило в эту правку (сознательно)

- Не трогал `event_occurrence_stats` как таблицу и её `increment()/decrement()` — сам факт, что кеш дрейфует при массовых отменах через `EventRegistrationsManagementController::cancel()` (прямой `DB::table()->update()` без вызова `EventOccurrenceStatsService`), это отдельная, более широкая проблема (влияет не только на дашборд), не в скоупе этой задачи.
- Не учитывал `event_occurrence_game_settings` (per-occurrence override вместимости) и `event_role_slots` — у организатора 260 в проверенных данных они не используются; если на проде у кого-то вместимость occurrence переопределена индивидуально, средняя загрузка будет считаться по вместимости на уровне события (небольшая неточность, не создающая пустых/противоречивых цифр — в отличие от исходного бага).
- 108.3%-превышение (без учёта резерва в знаменателе) — не менял, не относится к описанным симптомам.
